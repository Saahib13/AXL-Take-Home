Start backend-first, not UI-first.
The very first thing to build is one server function that takes a category + difficulty, pulls a few rows from Supabase, asks Gemini for a valid multiple-choice question in structured JSON, and returns it. If that works, the rest of the app is just product logic around it.
For this project, use the standard Gemini generateContent flow, not the newer Interactions API. Google’s docs currently say the Interactions API is still in beta and that generateContent remains the recommended path for stable deployments. Gemini also supports structured outputs with JSON schema/Zod, which is exactly what you want for clean question objects. (Google AI for Developers)

3. Create the Supabase project
Create a new Supabase project and get:

Do not start with vectors yet. Start with plain Postgres tables and normal filtering. Vectors are optional polish later.

5. Seed the dataset
This is where a lot of the project value comes from.
Create a data/corpus.json file with around 300 to 1000 rows. That is enough to feel substantial without wasting time.
Each row should be a compact knowledge card, for example:
{
  "title": "OpenAI released GPT-4",
  "category": "AI",
  "difficulty": 2,
  "summary": "GPT-4 is a large multimodal model introduced by OpenAI with stronger reasoning and safety performance than earlier GPT models.",
  "fact_1": "It followed GPT-3.5",
  "fact_2": "It supported multimodal input in some forms",
  "fact_3": "It was used in ChatGPT Plus",
  "tags": ["openai", "llm", "ai"],
  "source_label": "encyclopedic",
  "source_url": null,
  "is_current": false
}

Make categories balanced.
A good split:
25% AI
20% startups/business
20% internet culture
20% technology/history
15% current events / fresh topics
Then write a simple seed script that inserts these rows into corpus_items.
Important rule
Keep each record short and question-friendly. You are not building a document retrieval system. You are building a trivia corpus.
Definition of done
You can query rows by category and difficulty from Supabase.

7. Build the Gemini question generator in isolation
Now create lib/gemini/generateQuestion.ts.
Use the Google GenAI SDK and structured output. Gemini’s structured outputs support JSON schema and Zod in JavaScript, which makes it much easier to force a clean response format. (Google AI for Developers)
Your question schema
Make Gemini return exactly:
{
  questionText: string;
  options: [string, string, string, string];
  correctIndex: number;
  explanation: string;
  category: string;
  difficulty: number;
}

Your generation prompt should say
use only the supplied context
write one clear multiple-choice question
exactly 4 options
exactly 1 correct answer
no “all of the above”
no trick ambiguity
keep options similar in length
explanation must justify the correct answer using supplied context
output only valid JSON matching the schema
Also add validation
Use Zod to validate the model output. If validation fails:
retry once or twice
if it still fails, return a server error
Test it outside the app first
Create a script in scripts/test-gemini-question.ts that passes mock corpus context and prints the result.
Definition of done
From the terminal, you can generate a valid question object repeatedly.

8. Build the retrieval function from Supabase
Create lib/game/getQuestionContext.ts.
This function should:
accept category, difficulty, and excludeIds
query corpus_items
fetch 3 to 5 rows
avoid duplicates already used in the session
return compact context strings
At first, do plain SQL-style filtering:
match category
prefer matching difficulty
randomize a bit
limit 3–5 rows
Do not add embeddings yet unless you already have the core loop working. Supabase supports pgvector for embeddings and vector similarity, but that is an enhancement, not your starting point. (Supabase)
Definition of done
Given a category and difficulty, you can fetch usable context rows.

9. Combine retrieval + Gemini into one service
Create lib/game/createQuestionForSession.ts.
This function should:
inspect the session
choose category
choose difficulty
fetch corpus rows
call Gemini
save the generated question to session_questions
return the saved question
This is your most important function.
Inputs
sessionId
Outputs
question record with:
question text
options
question number
category
difficulty
prize amount
Definition of done
Given a session ID, the database gets a new session_questions row and your code returns it.

10. Build the API routes in this exact order
Route 1: POST /api/game/start
This should:
create a new game_sessions row
set status = active
set question number to 0
immediately call createQuestionForSession
return:
sessionId
first question
prize ladder
lifeline availability
Route 2: GET /api/game/[sessionId]
This should return current state:
session info
latest question
winnings
used lifelines
status
Useful for page refresh recovery.
Route 3: POST /api/game/[sessionId]/answer
This should:
accept selected option index
find the latest unanswered question
compare deterministically against stored correct_index
update the question row
update session winnings/status
return:
isCorrect
correctIndex
explanation
updatedWinnings
gameStatus
canAdvance
Route 4: POST /api/game/[sessionId]/next
This should:
verify previous question is answered or skipped
if game is still active and question count < 10:
generate next question
otherwise return final status
Route 5: POST /api/game/[sessionId]/lifeline/fifty-fifty
This should:
verify it has not been used
verify there is an active unanswered question
compute two removable wrong answers deterministically
mark session lifeline as used
return indexes to disable
Route 6: POST /api/game/[sessionId]/lifeline/ask-host
This should:
verify it has not been used
fetch current question
call Gemini to generate a short hint
store hint on question row
mark lifeline used
return hint text
Route 7: POST /api/game/[sessionId]/lifeline/skip
This should:
verify it has not been used
mark current question as skipped
mark session lifeline used
call next question generation
return next question
Definition of done
You can play a full game using only Postman, Thunder Client, or curl.

11. Add the optional dynamic behavior last
The take-home asks for dynamic behavior. The easiest guaranteed dynamic behavior is player adaptation:
stronger performance increases difficulty
repeated wrong answers lower or stabilize difficulty
categories rotate based on prior play
That already counts as adaptation based on user behavior.
If you want a stronger story, add one Today’s Bonus route that uses Gemini’s Google Search grounding. Gemini’s grounding with Google Search is designed for real-time information and can return citations. Also, Gemini 3 supports combining structured outputs with built-in tools like Google Search. (Google AI for Developers)
Build this only after the core game works.
Suggested route:
POST /api/game/[sessionId]/today-bonus
This route:
calls Gemini with Google Search grounding enabled
asks for one current-event trivia question in structured JSON
stores source metadata if you want to display it
Definition of done
One question in the game can be based on fresh web information.
